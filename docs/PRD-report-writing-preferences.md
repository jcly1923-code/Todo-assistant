# PRD：基于用户自写日报的语气库学习与生成润色

| 文档信息 |  |
|----------|--|
| 状态 | Draft v2（修正核心机制） |
| 产品模块 | 报告（日报 / 周报）+ 语气库管理 |
| 核心纠正 | 主输入是**用户自己写的日报文本**；系统从中**学习**并维护**语气库**；生成时以**语气库**为参考润色风格，而非简单拼接全文范文。 |

---

## 1. 产品定义（一句话）

用户提交**本人撰写**的日报（可多条、可更新）；系统在每次保存/更新后，**重新学习**用户的口头禅与表达习惯，形成并维护**语气库**；在生成新的日报、周报时，在**事实材料不变**的前提下，以语气库为参考**润色书写风格**。

---

## 2. 背景与问题

### 2.1 背景

当前 AI 根据待办事实生成报告，文风偏通用。用户实际写作中有稳定的**语气特征**（口头禅、句式、详略、段落习惯等），若仅依赖单次提示词，很难持续贴近。

### 2.2 要解决的问题

- 让 AI **从用户真实写过的日报里学习**表达方式，而不是让用户用抽象规则描述自己。
- **语气库**与「原始素材」分离：素材可增删改；每次变更后，**语气库同步更新**，保证后续生成用的是**提炼后的风格**，避免每次都塞超长原文进模型。
- 生成日报 / 周报时：**事实与结构仍严格来自待办与现有业务规则**；语气库只影响**怎么说**，不影响**说什么（事实）**。

### 2.3 非目标

- 不替代用户对事实的核对；**事实约束优先于语气**。
- 不承诺与某篇旧日报逐句一致（避免拟合过头与编造）。
- 本期不扩展到「任意文体写作助手」，聚焦**工作报告**场景。

---

## 3. 核心概念

### 3.1 用户输入：自写日报素材

- **定义**：由用户本人撰写、提交的日报正文（可多条，如按日期或按篇）。
- **用途**：作为**唯一或主要**的学习语料，供系统提取语气特征。
- **可选补充**（若产品需要降低冷启动难度）：允许极少量「补充说明」，但**不能替代**自写日报作为学习主来源（与本 PRD 主路径冲突时需以主路径为准）。

### 3.2 语气库（Tone Library）

- **定义**：从用户自写日报中**学习得到**的、结构化或摘要化的风格知识，用于指导后续生成时的润色。可包含但不限于：
  - 口头禅 / 常用衔接语
  - 句式偏好（长短句、列表偏好等）
  - 称谓与收尾习惯
  - 详略与分段习惯（在合规、脱敏前提下抽象描述）
- **更新规则（关键）**：**每次**用户**新增、编辑或删除**自写日报素材并保存成功后，触发**语气库重建或增量更新**（产品对外可表述为「已根据你的最新日报更新语气」）。
- **与素材的关系**：语气库是**提炼结果**；用户可查看「当前语气库摘要」（建议提供），便于理解与信任；**原始日报文本**的存储策略见隐私与数据章节。

### 3.3 生成时的使用方式

- 生成日报 / 周报时：先按现有逻辑组装**事实与结构**（待办、日期、补充事实等）。
- 在调用 AI 时，将**语气库**作为**风格参考**，要求模型在**不改动事实**的前提下润色表述。
- 若用户关闭「使用语气库」或尚未有足够素材学习，则回退为当前线上默认文风。

---

## 4. 用户故事

1. **作为用户**，我提交几篇**自己写的日报**，让系统学习我的语气。
2. **作为用户**，我修改或删掉其中一篇后，希望系统**自动更新**语气库，使后续生成跟上我最新的习惯。
3. **作为用户**，我在生成新的日报 / 周报时，希望文风**更接近我平时的写法**，而不是套话。
4. **作为用户**，我可以**关闭**语气润色，只要「事实型」正文。
5. **作为用户**，我可以**清空**所有素材与语气库，保护隐私。

---

## 5. 用户旅程（主路径）

```text
进入「语气学习」入口
  → 提交 / 编辑多份「自写日报」
  → 保存成功 → 系统学习 → 更新语气库（有明确成功反馈）
  → 在报告页生成日报或周报
  → 若开启语气库：生成结果在事实不变前提下贴近用户表达习惯
```

**与「割裂感」**：

- 主入口可放在**设置**或独立页「我的语气库」，但**报告页**需有短链：「先去提交几篇自写日报，生成会更像你」——避免用户只在报告页却找不到学习入口。

## 6. 功能需求（MVP）

### 6.1 语气学习素材管理

| ID | 需求 | 验收标准 |
|----|------|----------|
| F1 | 用户可新增多条自写日报 | 每条有标题或日期标识、正文、保存 |
| F2 | 用户可编辑、删除已有条目 | 保存后触发语气库更新 |
| F3 | 保存后触发学习 | 成功或失败有明确提示；失败可重试 |
| F4 | 总开关「生成时使用语气库」 | 关闭后生成不引用语气库 |

### 6.2 语气库

| ID | 需求 | 验收标准 |
|----|------|----------|
| T1 | 语气库随素材更新而更新 | 任意素材变更并保存后，语气库与「上次生成」一致 |
| T2 | 用户可查看语气库摘要（建议） | 展示提炼后的要点（非必须全文复述原始日报） |
| T3 | 一键清空 | 清空素材与语气库；生成恢复默认风格 |

### 6.3 报告生成

| ID | 需求 | 验收标准 |
|----|------|----------|
| G1 | 日报 / 周报均适用 | 开关开启且语气库可用时，两次生成都引用语气库润色 |
| G2 | 事实不被语气覆盖 | 与现有「严格依据事实材料」等规则一致，可测 |

### 6.4 隐私与提示

| ID | 需求 | 验收标准 |
|----|------|----------|
| P1 | 显著说明 | 提交内容为本人撰写；请勿包含敏感信息；用途为学习语气 |
| P2 | 数据归属 | 仅当前登录用户可读可写 |
| P3 | 删除权 | 用户可删除单条素材或全部；删除后重新学习语气库 |

## 7. 非功能需求

| 类型 | 要求 |
|------|------|
| 安全 | HTTPS；存储与访问控制；合规评估 |
| 性能 | 单次「学习」耗时可接受（可异步：保存后后台更新，完成前提示「语气库更新中」） |
| 可解释 | 用户知道「当前是否已使用语气库」 |

---

## 8. 技术方案要点（供研发，非最终实现）

- **学习流程**：素材变更 → 触发任务（同步或异步）→ 调用 LLM 或规则管线，将多篇日报**压缩为语气库表示**（结构化 JSON、或短摘要、或 embedding + 检索，由研发选型）。
- **生成注入**：在 `build_daily_prompt` / `build_weekly_prompt` 或 `ai_service` 层，将「语气库」以固定段落注入，明确 **仅润色、不编造事实**。
- **存储**：`user_daily_samples`（原文或摘要）+ `user_tone_profile`（语气库版本、内容、updated_at）；或合并表，需版本号便于排错与回滚。

---

## 9. 风险与对策

| 风险 | 对策 |
|------|------|
| 用户粘贴非本人内容 | 产品提示 + 用户协议；不保证效果 |
| 学习耗时 | 异步 + 状态提示 |
| 语气过强导致事实松散 | 提示词与评测强调事实优先；语气库仅作风格层 |
| Token 成本 | 语气库保持紧凑；原文不全文进每次生成 |

---

## 10. 成功指标（建议）

- 保存素材后 **语气库更新成功率**
- 开启语气库后，用户对生成稿的**编辑量**变化（若可埋点）
- **清空 / 关闭** 功能使用率（过高可能说明体验或信任问题）

## 11. 里程碑

| 阶段 | 范围 |
|------|------|
| **MVP** | 自写日报素材 CRUD + 保存后学习更新语气库 + 生成注入 + 开关 + 清空 |
| **V1.1** | 语气库摘要展示、异步学习进度、失败重试 |
| **V1.2** | 从润色稿 diff 辅助更新语气库（可选） |

---

## 12. 待决问题

1. 学习是否必须**异步**（影响「保存后立刻生成」的体验）。
2. 语气库是否对用户**展示全文**还是仅**摘要**。
3. 单用户素材条数与总字数上限（防滥用与成本）。
